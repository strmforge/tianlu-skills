# 天箓 Skills 概览

天箓 Skills 是一套中性的 Agent 工作流与产出方法系统。

它不是绑定某个模型、某个 Agent 运行时、某个项目、某个行业或某种个人习惯的 skill 包。它的目标是记录方法、标注适用域、维护方法之间的关系，并用证据决定方法是否应该保留、适配、组合、升格或废止。

天箓把方法当成一个循环：

```text
吸收有偏输入 -> 生成中立方法 -> 组合有偏变体 -> 再次吸收
```

最简定义是：承载吸收带作用域的有偏方法，生成抽取中立通用版本，创造再把中立方法重新组合成新的有偏版本。创出的版本经过实践后，再回到承的位置。这个过程是可无限组合的圆，不是一次性升格路径。

信号可以来自文档、事故、用户纠错、外部信息流、仓库、验法结果和实际经验。多数信号已经带着作用域偏向：它来自某个事件、运行时、项目、工具或任务。信号不能直接变成方法；它要先成为候选，再在 trigger、mechanism、evidence、failure mode、boundary 足够清楚后生成中立通用方法。中立不是空，而是可塑：它尚未固定到某个具体事件，所以还能被复用和重组。中立通用方法也不是终点，而是后续组合成运行时、项目或任务变体的稳定积木。变体天然带有新的作用域偏向；它如果保留来源、证据和边界，也可以再次成为新的有偏输入，所以方法生成是循环，不是一次性转换。

天箓不应该只长成 gate 库。基础法负责权限、证据、安全、持久化、工具和升格边界，但 Agent 还需要能做事的产出方法：电商运营、小说创作、自媒体运营、研究、分析、客服、规划等。`playbooks/` 放这些面向产物的方法卡；`laws/` 为它们提供权限、证据、安全和升格边界。

从中立方法回到有作用域的变体不是退化，而是循环的落地半圈：中立方法保证可复用，有作用域的变体保证可实用。变体只有在来源、作用域、证据和边界仍可检查时，才允许再次回流为新的候选。

所以有作用域的方法会在圆里出现两次：入口处是带着实践证据的有偏候选，出口处是由中立方法重新组合出的有偏变体。前者负责被吸收，后者负责重新落地。两者不是高低关系，而是循环中的不同相位；关键是标清来源、作用域、证据和边界。

这个循环是可组合的，不是单线流程。系统可以吸收许多带作用域的来源，生成可复用的中立中间形态，再把这些中立方法组合成许多新的作用域变体。新的作用域变体可以很有价值，但不能冒充普遍方法；只要来源、作用域、证据和边界仍然可检查，它就可以再次回流为新的候选。

这个循环是一个圆：有偏变体进入系统，中立生成抽出可复用的可塑性，组合再把这种可塑性用于新的具体场景。这里的“创”不是凭空生成权威方法，而是把中立方法重新组合成面向某个运行时、项目、组织、机器或任务的有偏版本。新版本天然带作用域；危险不在有偏，而在隐藏作用域、绕过中立生成和证据审查，却声称自己是普遍方法。

换成操作规则就是：承和创都是有作用域相位，生才是中立相位。系统保持通用性的方式，不是拒绝有偏变体，而是不让创出的变体因为“来自中立方法”就跳过回流，直接冒充普遍法。

相位约束见：[方法圆环](method-cycle.md)。

## 三字模型

```text
tiandao = 源
laws    = 基础法集
playbooks = 产出法集
index   = 序
```

### 源：tiandao

`tiandao/` 放元规则：方法如何进入系统、如何确定适用域、如何成熟、如何冲突、如何升格或废止。

这个目录名是有意选择的：它对齐仓库 owner 的子系统命名惯例，同时避免使用会暗示更大范围的源层名称。见：[Tiandao 命名决策](../tiandao-naming-decision.md)。

### 集：laws

`laws/` 放基础方法：证据、权限、安全、持久化、工具和升格。它们让产出过程可审计，但不是天箓的全部。

### 产出：playbooks

`playbooks/` 放面向产物的方法卡。这里收录帮助 Agent 做出有用成果的方法，例如电商运营、长篇创作、自媒体运营、研究简报、客服流程等。playbook 可以引用 laws，但不应该把每个产出步骤都写成 gate。

### 序：index

`index/` 放秩序：路由、依赖、作用域、冲突和关系图。没有秩序的方法集合只是堆，有秩序的方法集合才可查询、可组合、可治理。

## 其他层

- `adapters/`：面向具体运行时的打包和安装说明，例如 Codex、Claude Code 或未来运行时。
- `overlays/`：项目、组织、机器或个人的局部方法脉络。
- `incubator/`：新方法入口，不是垃圾桶。
- `eval/`：验法题和期望行为。

## 什么适合进入这里

适合：

- 能写清 trigger、mechanism、evidence、failure mode、boundary 的方法。
- 能帮助 Agent 创造有用产物，并能写清输入、输出、质量信号和人工确认点的 playbook。
- 可以诚实标注适用域的 Agent 工作流经验。
- 未来可能被不同运行时 adapter 使用的通用法。
- 虽然不通用但足够有价值、适合留在 overlay 的局部方法。

不适合：

- 没有证据的一次性观点。
- 放错位置的运行时专属说明。
- 出现在中性文件里的私有项目术语。
- 绕过 review 的安装指令。
- 本来只需要 playbook，却被写成压制产出的重 gate。

## 方法生成方向

- 承：把有作用域偏向的资讯或方法记录为候选，并标注来源和作用域。
- 生：从重复出现或证据充分的候选中抽出中立、可复用的方法，并移除或容纳局部偏向。
- 创：把中立通用方法重新组合成有作用域偏向的新版本，同时保留来源、证据和边界。
- 回流：有价值的作用域版本可以再次成为有偏候选，但必须保留可检查的来源、作用域、证据和边界。

这样通用性和多样性不是冲突的：通用方法不是终点，而是能生成许多安全变体的母体。为某个具体事件生成的版本天然带着那个事件的偏向；它可以很有用，但只有在偏向被标出、抽离或留在 adapter/overlay 之后，才能升成更通用的方法。这个过程不是只向上升格的梯子，而是有偏实践被承载、中立方法被生成、新的有偏实践被创造并再次回流的圆。

## 当前种子法

- Agent 编排：任务边界、委派、角色、共享状态。
- 浏览器与运行时：浏览器委托权限、运行时回执、远程控制。
- CI 与 DevOps 工作流：输入来源、事件内容污染、修复通道、扫描器证据、端点库存、策略即代码。
- 证据与验法：来源证据、replay、独立验证。
- 安装与供应链：不可信安装文本、持久化、依赖安装出口。
- MCP、工具与授权：权限边界、最小 allowlist、绑定授权、运行时输出处理。
- 记忆与学习：上下文工作集、持久记忆写入、生命周期、轨迹、信息流学习、skill 生成。

## 当前种子产出方法

- 电商运营：机会报告、商品页简报、标题变体、促销计划、客服回复草稿。
- 长篇创作：故事定位、人设、世界规则、冲突阶梯、章节大纲、连续性、修订。
- 自媒体运营：选题池、角度简报、脚本、平台变体、包装简报、发布日历、复盘。
- 调研到简报：证据表、假设图、风险清单、决策选项、下一步行动。
- 实验设计与迭代：变量矩阵、成功和停止判据、结果复盘、下一轮 backlog。
- 素材多形态改写：多格式素材包、来源到变体映射、一致性检查、审核包。
- 反馈转 backlog：反馈台账、问题簇、优先级队列、任务包、决策日志、闭环草稿。
- 价值主张与叙事定位：定位简报、价值主张阶梯、叙事屋、异议表、主张风险标记。
- 草案与证据包：可审阅草案、主张-证据表、待审队列、审批包、非结论声明。
- 受监管文档证据地图：文档清单、要求-证据映射、缺口与冲突表、待审队列、非结论声明。
- 项目工作流状态综合：任务/状态清单、工作流地图、完成证据清单、过期或未知列表、交接草稿。
- 销售管线运营规划：管线快照、机会台账、资格矩阵、交易风险图、下一步草案。
- 合同履约跟踪：义务台账、交付物跟踪、证据回执日志、缺口表、升级队列。
- 财务运营审阅：财务运营快照、收入/支出台账、预算差异表、现金流备注、异常队列。
- 创作者商业化运营：商业化地图、受众- offer 匹配表、赞助/联盟台账、披露清单、提案包。
- 客服质检审阅：客服 QA 评分卡、会话审阅台账、失误模式表、整改包、复检计划。
- 数据分析决策支持：数据字典、质量报告、指标定义表、描述性发现、实验读数、异常队列。
- 执行交接：可执行队列、审批表、阻塞日志、回退说明、证据回收清单。
- 学习运营规划：学习目标简报、概念图、学习日程、自测模板、错因表、进度快照和下一轮调整。
- 本地门店运营规划：门店快照、开闭店清单、库存观察表、交接草稿、评价回复草稿、POS 摘要和下一班计划。
- HR 招聘运营规划：角色 intake 简报、JD 草稿、结构化面试计划、评分卡模板、反馈汇总、onboarding 清单和 HR 交接包。
- 医疗照护协调规划：预约准备卡、症状时间线、用药清单、医生问题清单、随访追踪表、转诊状态板、照护分工表和照护交接包。
- 旅行行程运营规划：旅行简报、行程网格、交通方案表、住宿候选、活动候选、预算表、风险说明、备选方案和预订交接包。
- 租赁住房运营规划：租赁运营简报、房源/单元快照、申请清单、租客筛查证据包、维修摘要、租金提醒草稿队列、搬入/搬出清单和交接包。
